# Agent 工作流模板（共享）

本模板定义了 Review 阶段各审查 Agent 的通用工作流框架。每个审查 Agent 基于此模板执行，确保注意力隔离和输出格式一致性。

---

## ⚠️ 执行约束

本工作流被拆分为三个注意力阶段，需严格按阶段顺序执行。**严禁跨阶段预读文件**。

| 阶段 | 步骤 | 允许读取 | 禁止读取 |
|------|------|---------|---------|
| ① 纯审查 | 1-3 | proposal, apply_log, diff, 相关代码 | review_report, 其他 Agent 报告, report-format |
| ② 对抗验证 | 4-5 | 其他 Agent 报告, 代码 | report-format |
| ③ 格式输出 | 6-7 | report-format | — |

---

## 阶段 ①：纯审查（注意力阶段）

### 步骤 1：读取提案和实现日志

```
读取：
- .sdd-tdd/proposal.md
- .sdd-tdd/apply_log.md

提取：
- Spec 清单（所有 Spec 编号、描述、路径类型）
- 实现变更的文件列表
- Apply 阶段的发现和备注
```

**理解任务上下文**：
- 本次开发的目标是什么
- 采用了哪些技术方案
- 有哪些已知权衡

### 步骤 2：获取变更范围

```
执行：
git diff <base-commit>...HEAD

输出：
- 变更统计（N 个文件，+X / -Y 行）
- 变更文件列表
```

如果 `.sdd-tdd/.dev-flow-state.json` 存在，从中读取基准 commit；否则使用分支的第一个 commit。

### 步骤 3：逐文件审查

```
对每个变更文件：

1. 读取文件全文（不是只看 diff），理解上下文
2. 从你的专属领域视角审查：
   - <DOMAIN-SPECIFIC-CHECKS>
   - （由各 Agent 定义文件指定）
3. 标注发现的问题和严重程度
```

每个问题必须包含：
- 文件路径和行号
- 问题描述
- 具体证据（代码片段引用）
- 修复建议

---

**阶段①完成确认（必须回答）：**
- 我已读取：proposal ✓、apply_log ✓、diff ✓、代码文件 ✓
- 新发现问题（概要）：<列出每个新问题的文件:行号和一句话摘要>
- review_report.md：**尚未读取** ✓
- 其他 Agent 报告：**尚未读取** ✓
- report-format.md：**尚未读取** ✓
- 注意力隔离规则：未被违反 ✓

---

## 阶段 ②：对抗验证（可选，由主 Orchestrator 调度）

### 步骤 4：读取其他 Agent 的发现

本步骤由主 Agent（Orchestrator）按需调用。当某个 Agent 报告了 ERROR 级问题且仅被 1 个 Agent 标记时：

```
读取：
- 其他 Agent 的报告（由 Orchestrator 指定）
- 相关代码文件

任务：
尝试反驳指定 finding，输出结构化的反驳结果
```

### 步骤 5：输出反驳结果

```json
{
  "refuted": true | false,
  "finding_id": "<被反驳的 Finding ID>",
  "reason": "<反驳理由，必须引用具体代码证据>",
  "evidence": "<代码片段引用>"
}
```

注意：反驳失败是正常情况。如果找不到反驳证据，必须诚实地承认反驳失败。

---

## 阶段 ③：格式输出

### 步骤 6：读取报告格式规范

```
读取：agents/report-format.md

严格遵循格式要求，不允许偏差。
```

### 步骤 7：写入结果

```
输出到：.sdd-tdd/review-<agent-name>.json

格式：严格 JSON，无 Markdown，无额外文字
```

## 通用规则

### 关于证据

每个发现的问题都必须有：
1. **具体位置**：文件路径 + 行号
2. **代码引用**：粘贴相关代码片段（10 行以内）
3. **影响分析**：如果不修复，会导致什么后果

没有证据的 finding 是无效的。

### 关于严重性标定

- **ERROR**: 必须修复。不修复会导致 bug / 安全漏洞 / 数据丢失 / 系统不可用
- **WARN**: 建议修复。当前可能不影响功能，但存在风险或不符合最佳实践
- **INFO**: 信息性观察。记录改进建议或值得了解的点

不要为了凑数而抬高或压低严重性。

### 关于注意力隔离

跨阶段预读会严重影响审查质量。常见越界模式：

- ❌ 在阶段①就读取上一轮的 review_report.md → 被旧问题锚定，无法发现新问题
- ❌ 在阶段①就读取其他 Agent 的报告 → 注意力被其他维度稀释
- ❌ 在阶段①-②就读取 report-format.md → 注意力分散到格式细节而非代码质量

**正确做法**：
- ✅ 阶段①只从自己的领域视角审查
- ✅ 阶段②仅在 Orchestrator 调度时读取指定材料
- ✅ 阶段③才读取格式规范

### 关于对抗验证的执行

当被调度为辩护 Agent 时：
- 你的任务是**尝试反驳**，而不是**尝试证明**
- 保持怀疑论者的心态：找证据说明这个问题不存在或严重性被高估
- 如果找不到反驳证据，诚实承认反驳失败
- 引用具体代码作为反驳依据，不允许空泛地说"看起来没问题"
