# Agent 工作流模板（共享）

本模板定义了 Review 阶段各审查子 agent 的通用工作流框架。每个审查子 agent 基于此模板执行。

**注意力隔离**：每个审查子 agent 是独立上下文，由主 Claude 用 Agent 工具派发，天然不读取其他 reviewer 的输出。无需"严禁预读"的自律表——隔离由执行模型保证。

---

## 工作流

### 步骤 1：读取提案和实现日志

```
读取：
- .sdd-tdd/proposal.md
- .sdd-tdd/apply_log.md

提取：
- Spec 清单（所有 Spec 编号、描述、路径类型）
- 实现变更的文件列表
- Apply 阶段的发现和备注
```

**理解任务上下文**：
- 本次开发的目标是什么
- 采用了哪些技术方案
- 有哪些已知权衡

### 步骤 2：获取变更范围

```
执行：
git diff <base-commit>...HEAD

输出：
- 变更统计（N 个文件，+X / -Y 行）
- 变更文件列表
```

如果 `.sdd-tdd/.dev-flow-state.json` 存在，从中读取基准 commit；否则使用分支的第一个 commit。

### 步骤 3：逐文件审查

```
对每个变更文件：

1. 读取文件全文（不是只看 diff），理解上下文
2. 从你的专属领域视角审查：
   - <DOMAIN-SPECIFIC-CHECKS>
   - （由各子 agent 定义文件指定）
3. 标注发现的问题和严重程度
```

每个问题必须包含：
- 文件路径和行号
- 问题描述
- 具体证据（代码片段引用）
- 修复建议

### 步骤 4：读取报告格式规范并写入结果

```
读取：agents/report-format.md
严格遵循格式要求，不允许偏差。

输出到：.sdd-tdd/review-<agent-name>.json
格式：严格 JSON，无 Markdown，无额外文字
```

## 通用规则

### 关于证据

每个发现的问题都必须有：
1. **具体位置**：文件路径 + 行号
2. **代码引用**：粘贴相关代码片段（10 行以内）
3. **影响分析**：如果不修复，会导致什么后果

没有证据的 finding 是无效的。

### 关于严重性标定

- **ERROR**: 必须修复。不修复会导致 bug / 安全漏洞 / 数据丢失 / 系统不可用
- **WARN**: 建议修复。当前可能不影响功能，但存在风险或不符合最佳实践
- **INFO**: 信息性观察。记录改进建议或值得了解的点

不要为了凑数而抬高或压低严重性。

### 关于对抗验证的执行

当被主 Claude 派发为对抗验证子 agent（adversarial-verifier）时：
- 你的任务是**尝试反驳**，而不是**尝试证明**
- 保持怀疑论者的心态：找证据说明这个问题不存在或严重性被高估
- 如果找不到反驳证据，诚实承认反驳失败（verdict=confirmed）
- 引用具体代码作为反驳依据，不允许空泛地说"看起来没问题"
- 输出单一 `verdict: refuted|confirmed|uncertain`
