# SDD-TDD 方法论

## 核心理念

SDD-TDD (Spec-Driven Development + Test-Driven Development) 是将规格驱动开发与测试驱动开发相结合的结构化开发方法论。

### 为什么需要 SDD-TDD？

**单独的 TDD 问题：**
- 测试可能验证错误的行为（需求理解错误）
- 缺乏整体架构视角
- 容易陷入局部优化

**单独的 SDD 问题：**
- 规格写了没人看
- 规格与实现脱节
- 无法验证规格是否被正确实现

**SDD-TDD 的解决方案：**
1. **规格即测试**：每个 Spec 都转化为可执行的测试用例
2. **实现受约束**：TDD 循环强制执行规格中定义的行为
3. **验证闭环**：测试通过 = 规格被正确实现
4. **审查对齐**：Review 阶段检查测试是否覆盖规格

## 三层相似模式

SDD-TDD 在三个粒度上有相似的工作模式（注意：是模式相似，非递归调用）：

```
整体流程：
  explore → proposal → apply → review → archive

每个 Phase：
  理解输入 → 设计方案 → 执行 → 验证 → 输出

每个 Spec：
  定义行为 → 写测试 → 实现 → 验证通过

每个 TDD 循环：
  明确目标 → 写失败测试 → 最小实现 → 重构
```

这种模式一致性确保了可预测性。

## 决策前置原则

### 传统开发
```
需求 → 写代码 → 遇到问题 → 临时决策 → 继续写代码
```

### SDD-TDD
```
需求 → explore（理解问题）→ design（做决策）→ apply（执行决策）
```

**好处：**
- 减少实现过程中的返工
- 决策可追溯（有记录）
- 多人协作时减少冲突

## 对抗验证原理

### 问题背景

代码审查中，审查者可能因为以下原因产生误报：
- 信息不完整
- 注意力分散
- 理解偏差

### 解决方案

**对抗验证（Adversarial Validation）**：对每个严重问题，分配一个独立的"反驳者"尝试证明问题不存在。

```
审查者提出 finding
    ↓
反驳者读取代码 + 追踪调用链
    ↓
形成判断：
  - refuted（反驳成功）→ 移除 finding
  - confirmed（反驳失败）→ 升级为 Must-Fix
  - uncertain（不确定）→ 人工判断
```

### 为什么不直接让审查者自我纠正？

- **认知偏见**：人们倾向于确认自己的初始判断
- **职责分离**：审查者专注于发现问题，反驳者专注于验证问题
- **提高质量**：双重检查机制显著降低误报率

## 知识归档价值

### 传统问题
- 每次开发都"从零开始"
- 重复犯同样的错误
- 新人学习成本高

### SDD-TDD 解决方案
1. **强制归档**：流程最后一步是归档（不是可选项）
2. **结构化输出**：统一的归档格式，便于检索
3. **历史索引**：`openspec-archive-index.md` 提供快速查找
4. **规格演化**：归档的 spec 成为未来项目的参考

### 归档内容
- 问题定义
- 方案选择及理由
- 关键决策及其原因
- 经验教训
- 完整的 spec 文件

## 适用场景

### 适合使用 SDD-TDD

✅ **复杂功能开发**
- 涉及多个模块
- 有多种实现方案
- 需要长期维护

✅ **团队协作项目**
- 多人分工
- 需要代码审查
- 需要知识传递

✅ **关键业务逻辑**
- 不允许出错
- 需要可追溯性
- 需要严格测试

### 不适合使用 SDD-TDD

❌ **快速原型 / Hackathon**
- 时间压力极大
- 目标是验证想法
- 代码质量次要

❌ **简单 CRUD 操作**
- 模式明确
- 决策很少
- 过度工程化

❌ **紧急修复**
- 问题明确
- 需要在几分钟内修复
- 不需要设计讨论

## 成功因素

### 1. 严格遵守流程

不跳阶段，不做例外。流程的价值在于**一致性**和**可预测性**。

### 2. 文档及时更新

- explore 阶段：及时记录发现
- design 阶段：及时记录决策
- archive 阶段：不省略、不敷衍

### 3. 审查认真投入

- Reviewer：真正理解代码，不是走过场
- Adversarial Verifier：真正尝试反驳，不是形式化
- 不要为了"和谐"而降低标准

### 4. 持续改进

- 根据项目实际情况调整 Spec 粒度
- 根据团队反馈优化流程
- 将失败案例加入归档

## 常见陷阱

### 陷阱 1：过度规格化

❌ **错误做法**
- 每个函数都写详细 spec
- spec 比代码还长
- 事无巨细地描述实现细节

✅ **正确做法**
- spec 关注行为，不关注实现
- 聚焦核心业务逻辑
- 简单的工具函数可以简化或跳过

### 陷阱 2：TDD 机械化

❌ **错误做法**
- 为了覆盖率而写测试
- 测试不验证真实行为
- 忽视测试质量

✅ **正确做法**
- 测试应该失败如果行为不正确
- 测试应该是可执行的规格
- 删除无用测试比添加无用测试更重要

### 陷阱 3：对抗验证形式化

❌ **错误做法**
- 随便写两句就标记为 refuted
- 不追踪调用链
- 为了完成而完成

✅ **正确做法**
- 真正读取相关代码
- 追踪上下游
- 诚实报告结果（refuted 或 confirmed）

### 陷阱 4：归档敷衍

❌ **错误做法**
- "完成"两个字就归档
- 不记录决策理由
- 不保存 spec 文件

✅ **正确做法**
- 详细记录问题和方案
- 解释为什么选择这个方案
- 保存完整的 spec，移入 archive

## 量化指标

### 流程效率

| 指标 | 目标 | 测量方法 |
|------|------|---------|
| 阶段完成率 | 100% | 每个任务完成所有阶段 |
| Spec 覆盖率 | ≥ 90% | spec-tracker.py 检查 |
| 测试通过率 | 100% | 全量测试通过 |
| Must-Fix 清零率 | 100% | review 后无 Must-Fix |

### 代码质量

| 指标 | 目标 | 测量方法 |
|------|------|---------|
| 回归测试失败率 | 0% | apply 阶段全量测试 |
| 审查误报率 | < 20% | 对抗验证 refuted / total ERR |
| 返工率 | < 10% | 回退到 design 的次数 / 总任务数 |

### 知识积累

| 指标 | 目标 | 测量方法 |
|------|------|---------|
| 归档完整率 | 100% | 每个任务都有归档 |
| spec 归档量 | 持续增长 | openspec-archive-index.md 条目数 |
| 重复决策率 | 下降 | 相同类型的决策次数减少 |

## 与其他方法论的关系

### 与 TDD 的关系

SDD-TDD **包含** TDD，但将 TDD 约束在 apply 阶段。TDD 是 SDD-TDD 的执行机制，而非设计机制。

### 与 BDD 的关系

SDD-TDD 的 Spec 格式类似 BDD 的 "Given-When-Then"，但更强调：
- 技术细节（文件位置、参数类型）
- 测试映射（每个 spec 对应具体测试函数）
- 覆盖率验证（spec-tracker 工具）

### 与敏捷的关系

SDD-TDD **兼容**敏捷，但节奏不同：
- 敏捷：2 周 sprint，快速迭代
- SDD-TDD：单任务完整流程，可能需要数小时到数天

建议在敏捷框架中使用 SDD-TDD 处理**高复杂度任务**。

## 总结

SDD-TDD 的核心价值：

1. **减少返工**：决策前置，避免实现中发现设计问题
2. **提高质量**：TDD + 多角色审查 + 对抗验证
3. **知识积累**：强制归档，避免重复犯错
4. **可预测性**：三层相似模式，流程一致

记住：**流程是手段，不是目的**。如果流程成为负担，说明需要调整。
